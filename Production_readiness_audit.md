# OpenUI — Production Readiness Audit

**Reviewed as:** full working tree at `Openui-main` (Electron desktop app, package version `7.0.4`)
**Standard applied:** ready-for-real-users production product, not an MVP
**Out of scope (per engagement instructions):** marketing site / landing pages / demo-request flows, "pro"/roadmap features not yet built, API key & secrets configuration (judged as if correctly configured server-side)
**Method:** direct source review (not README/ARCHITECTURE.md claims) — every finding below is traced to a specific file and, where useful, a line number.

---

## 1. Architecture & Flow Summary

This repository actually contains **four separate codebases**, only one of which is the shipping product.

**A. The real product** — `src/main` (Electron main process, TypeScript) + `src/renderer` (React UI) + `src/preload`. This is "OpenUI", a Windows/macOS desktop menu-bar assistant that signs the user in (Google OAuth via system browser, or a zero-friction anonymous guest session), routes chat through a Supabase Edge Function that holds the LLM keys server-side, and lets the model call a set of OS-automation tools (open apps, search files, control the calendar, move the mouse/keyboard, read/write files inside the home folder, control a headful Playwright browser) with a filesystem sandbox and a human-in-the-loop approval gate on state-changing actions. Billing runs through Stripe Checkout/Portal opened in isolated Electron windows, with Stripe webhooks writing the authoritative tier into Supabase `app_metadata` and the client re-syncing every 5 minutes. Conversations, users, and subscription cache are persisted locally in SQLite (`better-sqlite3`). This is the codebase that `package.json`, `electron.vite.config.ts`, `.github/workflows/release.yml` and the auto-updater all build and ship.

**B. `supabase/functions/*`** — six Deno Edge Functions (`chat-proxy`, `check-subscription`, `create-checkout`, `customer-portal`, `stripe-webhook`, `waitlist`) that are the *actual* backend for (A). Only `chat-proxy` is exercised by the primary shipped flow.

**C. A completely disconnected legacy prototype** — root `main.py`, `core/`, `tools/`, `ui/`, `voice/`, `models/` (~11,500 lines of Python). This is an older, local-only, PyQt5 + Ollama version of "OpenUI v0.1.0" with its own agent loop, its own tool registry, its own overlay UI. **Nothing in `src/` imports or shells out to any of this** — it is dead weight sitting next to the real app.

**D. Two more disconnected server attempts** — `server/` (a Python service with three different, mutually-inconsistent entry points: `main.py`, `app.py`, `fastapi_app.py`) and `api/` (a separate, smaller FastAPI skeleton with auth + rate-limit middleware). Neither is wired into the shipping product by default — `src/main/serverClient.ts` only talks to `server/` if a developer sets `VITE_SERVER_URL`, which ships unset. `api/` isn't referenced by any Dockerfile, docker-compose file, or CI job at all.

The **working end-to-end flow**, as actually implemented in (A)+(B), is: app launches → `ensureGuestSession()` silently mints a Supabase anonymous session so the app is usable with zero setup → user types or speaks a request → `handleChat()` in `src/main/agent.ts` optionally builds a multi-step plan (`planner.ts`), asks for one approval, then loops the model through `chat-proxy` → the model emits a `{"tool": ..., "args": ...}` JSON object → `executeTool()` runs it (with a path-safety/permission/HITL gate for anything that touches disk, mouse, keyboard or the browser) → the result is fed back to the model → the loop continues until a natural-language answer is produced → the turn is persisted to SQLite and to a "training store" used by a weekly local self-improvement job. Subscription state, once a Stripe checkout completes, flows Stripe → `stripe-webhook` Edge Function → Supabase `app_metadata.tier` → `syncSubscriptionStatus()` → local cache → renderer.

## 2. Project Completeness Rating

### 60 / 100 — Grade C-

This is graded strictly as "can I put this in front of paying users today," per your instructions — not against the bar of a typical early-stage repo (against *that* bar it would score noticeably higher).

**What earns real credit.** The parts of this app that were clearly the primary focus — auth, billing, the Electron security boundary, and the tool-execution sandbox — are built to a standard well above a typical MVP. Concrete examples: OAuth is routed through the system browser specifically to work around Google's embedded-webview block (`src/main/auth/authWindow.ts`); OS commands are parameterized through child-process environment variables instead of string concatenation, closing off PowerShell/AppleScript injection (`src/main/tools.ts`); every LLM-supplied filesystem path is resolved and checked against a sensitive-path blocklist and a home-directory boundary before use (`src/main/fs/pathSafety.ts`); a compromised renderer cannot escalate its own subscription tier because the tier is clamped server-side against the verified entitlement (`clampTierToEntitlement` in `src/main/stripe/pricing.ts`); the Electron CSP, sandboxing, and navigation lockdown in `src/main/index.ts` are genuinely thorough. The core chat loop, once it reaches the model, is solid engineering.

**What disqualifies it from "ready."** Several of the product's headline capabilities — the ones with dedicated UI, dedicated hotkeys, and dedicated usage-limit tracking already built for them — cannot work for a single real, shipped user, independent of how correctly you configure your API keys. This isn't a "you forgot to set a key" problem (explicitly out of scope per your instructions); it's that the *code path* to deliver a correctly-configured key to where it's needed doesn't exist for these features, unlike chat, which solved this correctly. Voice input/output, Pro-tier "cloud vision" screen reading, the AI interviewer, the Figma design-review workflow, and GitHub PR review all fall over the same cliff (see 🔴 URGENT #1–#5). On top of that, three Supabase Edge Functions have a genuine broken-access-control bug on billing-adjacent endpoints, the repository is roughly 40% dead/duplicate/orphaned code that actively risks a future contributor editing the wrong file, and there is effectively no automated test coverage enforced anywhere in CI.

A product where a paying Pro customer's first click on the feature they upgraded for ("Cloud vision") throws a raw API error is not production-ready, no matter how well-built the surrounding shell is. That combination — a genuinely strong core wrapped around several completely non-functional advertised features and a real security gap — is what puts this at a C- rather than higher.

## 3. Actionable Task Breakdown

### 🔴 URGENT — Critical blockers, security vulnerabilities, or core functionality that is entirely broken

**✅ COMPLETED — 1. Voice input and output cannot work for any shipped user.**
`src/main/voice.ts` — `transcribeWithWhisper()` (line 47) and `synthesizeSpeech()` (line 124) both call the OpenAI/ElevenLabs SDKs directly using `process.env.OPENAI_API_KEY` / `process.env.ELEVENLABS_API_KEY`. Compare this to `chat`, which correctly proxies through the `chat-proxy` Supabase Edge Function so the key never has to reach the client. Voice has no such proxy. Checking `electron.vite.config.ts`'s `bake()` allowlist confirms `OPENAI_API_KEY`/`ELEVENLABS_API_KEY` are *deliberately not* inlined into the packaged binary (correctly, for security), there is no `dotenv` load anywhere in `src/main` (confirmed by search), and `SettingsModal.tsx` has no field where a user could paste their own key. The result: `process.env.OPENAI_API_KEY` is `undefined` for every real installed copy of the app, so every voice request throws `"OPENAI_API_KEY is not set"`. This is not a missing-configuration issue — there is no code path by which a correctly-configured key would ever reach this function for a shipped user. A full UI (mic hotkey, `RecorderUI.tsx`, monthly/daily voice-minute tracking in `voice_usage` table) was built around a backend that cannot run. **Fix:** add a `voice-proxy` (or two: transcribe/synthesize) Edge Function mirroring `chat-proxy`'s pattern — hold the key server-side, accept the audio/text over HTTPS with the user's Supabase token, enforce the already-implemented `voice_usage` limits, return the transcript/audio — and repoint `voice.ts` at it.
> **Done:** Added `supabase/functions/voice-proxy/index.ts` (verifies the caller's Supabase token, holds `OPENAI_API_KEY` + optional `ELEVENLABS_API_KEY` server-side, exposes `{ action: 'transcribe' | 'synthesize' }`). A new shared client `src/main/edgeFunctions.ts` centralises token/URL resolution; `transcribeWithWhisper()` / `synthesizeSpeech()` in `voice.ts` now call the proxy through it instead of the OpenAI/ElevenLabs SDKs (both SDK imports removed). Per-tier `voice_usage` limits stay enforced as before in `voice.ts`. Deploy + secret docs added to `supabase/functions/README.md` and `.env.example`. Typecheck + build pass.

**✅ COMPLETED — 2. Pro/Enterprise "cloud vision" (`read_screen`) is broken for paying customers.**
`src/main/tools.ts`, inside `read_screen()` (~line 772): `new Anthropic({ apiKey: process.env.ANTHROPIC_API_KEY ?? '' })` — called directly from the client for the pro/enterprise branch. Same root cause as #1: this key is never baked, never proxied, and has no Settings-UI input. "Cloud vision (screen understanding)" is listed as a named Pro-tier feature in `src/main/stripe/pricing.ts` — a customer who pays $19 for it will get a tool-execution error the first time the agent tries to use it. **Fix:** proxy this through `chat-proxy` (it already accepts Anthropic-style image content blocks) or a dedicated `vision-proxy` function.
> **Done:** `read_screen()` in `tools.ts` now sends the screenshot as an Anthropic-style image block through `chat-proxy` via `callChatProxyText()` (in `edgeFunctions.ts`), with a tier-scoped `modelKey` the proxy clamps to the caller's entitlement. The direct `new Anthropic({ apiKey: process.env.ANTHROPIC_API_KEY })` client and the `@anthropic-ai/sdk` import were removed from `tools.ts`. No LLM key reaches the client. Typecheck + build pass.

**✅ COMPLETED — 3. The AI Interviewer feature is non-functional.**
`src/main/interviewer.ts` (line 127-128): `if (!apiKey) throw new Error('ANTHROPIC_API_KEY is required for the AI interviewer.')` with no cloud-proxy branch. This is a whole "Phase 12" feature with its own IPC surface (`startInterview`, `sendInterviewAnswer`, transcript/status events in `src/preload/index.ts`) that will throw on first use for every shipped user. **Fix:** same proxy pattern, or remove the entry points from the UI until it's wired up.
> **Done:** `generateQuestion()` in `interviewer.ts` now calls Claude through `chat-proxy` via `callChatProxyText()` with a tier-scoped `modelKey`; the `ANTHROPIC_API_KEY` check/throw and the `@anthropic-ai/sdk` import were removed. Its speech-to-text and text-to-speech steps go through the same `voice-proxy` as URGENT #1, so the whole feature runs with no client-side keys. Typecheck + build pass.

**✅ COMPLETED — 4. Figma design-review workflow is non-functional.**
`src/main/figma.ts` (lines 331 and 478) requires `ANTHROPIC_API_KEY` and `FIGMA_TOKEN` directly from `process.env`, with no proxy and no Settings UI. The chat-triggered "designer workflow" (`DESIGNER_SYSTEM_PROMPT` in `agent.ts`) will fail on the first `export_figma_frames` call for every real user. **Fix:** same proxy pattern for the Anthropic call; for `FIGMA_TOKEN` specifically (a personal, per-user credential, unlike an LLM key), the more appropriate fix is a Settings-UI field so users supply their own — there's no reason to proxy a token that's inherently user-owned.
> **Done:** `export_figma_frames()` in `figma.ts` now runs Vision through `chat-proxy` via `callChatProxyText()` (direct Anthropic client + import removed). The Figma personal-access token is now a per-user Settings value: `getFigmaToken()` reads the `figma_token` setting (env `FIGMA_TOKEN` kept only as a dev fallback), and a password-style "Figma" field was added to `SettingsModal.tsx` that persists via the existing `set-setting` IPC. Tool-schema/`.env.example` docs updated. Typecheck + build pass.

**5. GitHub PR review and GitHub-issue-sourced autonomous coding are non-functional; local-task autonomous coding is fine.**
`src/main/github.ts` (line 56): `process.env.GITHUB_TOKEN` read directly, no proxy, no Settings UI, not in the `electron.vite.config.ts` bake list. This breaks two things: the "review my PRs" chat command (`PR_REVIEW_SYSTEM_PROMPT` in `agent.ts`, which calls `list_open_prs`/`get_pr_diff`/`post_pr_comment`), and the GitHub-issue task source for Autonomous Coding Mode (`fetchGitHubIssues()` in `src/main/tasks.ts`). To be fair to what *does* work: Autonomous Coding Mode's default local `todo.json` task source (`src/main/tasks.ts`, `src/main/sandbox.ts`, `src/main/autonomous.ts`) only needs the model call, which is properly proxied — that path is genuinely functional. **Fix:** add a Settings-UI field for a personal-access token (same reasoning as Figma's token above), since forcing every user's GitHub calls through your own server would need per-user OAuth anyway.

**6. Broken object-level authorization on three billing-adjacent Supabase Edge Functions.**
`supabase/functions/create-checkout/index.ts`, `check-subscription/index.ts`, and `customer-portal/index.ts` all read `userId` straight out of the unauthenticated-looking request body (`const { userId, ... } = await req.json()`) and use it directly, **without** ever calling `supabase.auth.getUser(token)` to confirm the caller's JWT actually belongs to that `userId`. Compare this to `chat-proxy/index.ts`, which does this correctly (`supabase.auth.getUser(token)` before trusting anything). Practical impact: any authenticated caller — including a zero-friction anonymous guest account, which `ensureGuestSession()` mints with no user action at all — can pass an arbitrary Supabase user UUID to `check-subscription` and read that person's subscription tier, status, and Stripe customer ID, or open a Stripe billing-portal/checkout session tagged to someone else's account. Exploitation requires knowing/guessing a target's UUID (not sequential, so not trivially enumerable), which lowers practical risk, but this is still a genuine broken-auth bug on payment infrastructure and the fix is small. **Fix:** in each of the three functions, extract `user.id` from the caller's verified JWT (same one line `chat-proxy` already uses) and use *that* instead of, or validated against, the client-supplied `userId`.

**7. `server/fastapi_app.py` cannot run — hard syntax error.**
Line ~40: `await redis_client.ping()` sits at module top level, outside any `async def`. `await` outside a function is a Python `SyntaxError`; this file cannot be imported, let alone started, as-is. It isn't currently referenced by the `server/Dockerfile` CMD (which runs `main:app`), so it's not an active blocker today — but it's shipped as if it were working code. **Fix:** either finish and fix it, or delete it.

**8. The `server/` multi-user backend, if anyone deploys it, silently serves a fake API.**
Three competing, inconsistent implementations exist for the same conceptual service: `server/main.py` (the one the `Dockerfile` actually runs, and the one the root `docker-compose.yml` wires up as the `api` service on port 8000) has a `/chat` endpoint and a `/ws/{user_id}` WebSocket that both **literally echo the input back** (`reply = f"[echo] {body.message}"`, `# TODO Task 2: route body.message through SessionManager → TaskRouter → Agent`). `server/app.py` (Flask, tier enforcement + Stripe webhook) is more complete but its bearer-token check is a stub that accepts any non-empty token as valid for any user (`# TODO: Verify token is valid (validate Supabase JWT). For now, just check it exists`). `server/fastapi_app.py` (see #7) doesn't run at all. Meanwhile `server/docker-compose.yml` health-checks port 5000 while the `Dockerfile` it builds actually exposes/serves port 8000 — the compose file and Dockerfile disagree with each other. **Fix:** since the shipped Electron app does not need `server/` by default (it works entirely through the Supabase Edge Functions), the pragmatic move is to either fully finish and consolidate `server/` into one real implementation matching the WebSocket contract `src/main/serverClient.ts` already expects, or delete `server/` and the root `docker-compose.yml` entirely so nobody accidentally deploys a fake backend.

### 🟠 HIGH — Major bugs, missing essential user flows, or severe stability/performance issues

**1. There is effectively no automated test coverage on the highest-risk code.**
Only three test files exist in the whole TypeScript codebase (`src/main/fs/pathSafety.test.ts`, `src/main/stripe/pricing.test.ts`, `src/renderer/src/lib/telemetry.test.ts`) against roughly 18,000 lines of TypeScript. `src/main/agent.ts` (~1,200 lines — the tool-call parser, the `StreamGate` that keeps raw tool JSON off the user's screen, the HITL/plan-approval state machines) has zero tests. `src/main/tools.ts` (~1,670 lines — every OS-automation action the agent can take) has zero tests. Every database repository under `src/main/database/repositories/` has zero tests. For a product whose core value proposition is "an LLM can control your computer," the code that decides what the LLM is and isn't allowed to do is the single most important thing to have regression tests on, and currently doesn't.

**2. The tests that DO exist aren't run in CI.**
`.github/workflows/pr-check.yml` runs `npm run typecheck` and `npm run build` on every PR, but never `npm test` / `npm run test`even though the script is defined in `package.json`. A PR could break `pathSafety.ts` — literally the function standing between an LLM and unrestricted filesystem access — and the existing `pathSafety.test.ts` would never run to catch it. This is a one-line fix (`- run: npm test` added to `pr-check.yml`) that meaningfully changes the safety story.

**3. `api/` is a fully-built, fully-tested, entirely disconnected FastAPI service.**
`api/main.py`, `api/dependencies.py`, `api/middleware/auth.py`, `api/middleware/rate_limit.py`, plus a real test suite (`api/tests/test_auth.py`, `api/tests/test_rate_limit.py`) implement JWT auth middleware and Redis-backed sliding-window rate limiting to a reasonable standard — but its only two business endpoints are stubs (`api/main.py` lines ~78-91: `"message": "chat handler not yet implemented"` / `"voice handler not yet implemented"`), and nothing in the repo — no Dockerfile, no docker-compose service, no CI job — ever builds or runs it. A contributor (human or AI) reading this repo could reasonably mistake `api/` for the real backend, given it has better test hygiene than anything else in the Python tree. **Fix:** wire it in and finish it, or delete it.

**4. ~11,500 lines of orphaned legacy Python roughly double the apparent size of the codebase.**
Root `main.py`, `core/` (agent.py, router.py, config.py, memory.py, task_planner.py, task_executor.py, vision_loop.py, setup.py, helpers.py), `tools/`, `ui/`, `voice/`, `models/` implement a complete, separate, older PyQt5+Ollama version of "OpenUI" that nothing in `src/` calls into (confirmed by searching `src/` for any reference to `main.py`, `core/agent`, or a Python subprocess spawn — none exist). This isn't neutral clutter: it has files with near-identical names to the real ones (`core/agent.py` vs. `src/main/agent.ts`, `tools/registry.py` vs. `src/main/tools.ts`, `voice/stt.py`/`voice/tts.py` vs. `src/main/voice.ts`), which is a real risk that a future engineer — or an AI coding assistant asked to "fix the voice bug" — edits the dead copy instead of the live one. **Fix:** delete it, or move it to an explicitly-labeled archive branch/tag outside the working tree.

**5. Duplicate, dead local-database implementation.**
`src/main/database/db.ts` (186 lines) defines its own `users`/`conversations`/`messages` schema and CRUD functions (`dbGetUser`, `dbUpsertUser`, `dbSaveMessage`, etc.) that are never imported by anything except the file itself — confirmed by search. The real, actually-used path is `src/main/database/index.ts` → `init.ts` + `schema.ts` + `migrations.ts` + `repositories/*.ts`. `db.ts`'s own schema is missing the `refresh_token`/`token_expires_at` columns the real auth flow depends on, so it couldn't even serve as a drop-in replacement if someone tried to switch to it. **Fix:** delete `db.ts`.

**6. No real migration path for the local SQLite schema.**
`src/main/database/migrations.ts` registers an empty `migrations: Migration[] = []` array; every table is created via `CREATE TABLE IF NOT EXISTS` in `schema.ts`. That's fine for a brand-new install, but there is currently no exercised mechanism to alter an *already-installed* user's database when the schema changes after v1 ships (e.g., adding a column to `users`). **Fix:** land at least one real migration through this system before your first schema change post-launch, so the untested path isn't discovered for the first time under production pressure.

**7. Free-tier telemetry logs the wrong model name.**
`src/main/agent.ts`, `modelForTier('free')` returns `process.env.OLLAMA_MODEL ?? 'llama3:8b'`, and this value is what gets sent to PostHog via `trackEvent(Events.CHAT_MESSAGE_SENT, { model, ... })` and `CHAT_RESPONSE_RECEIVED`. But free-tier chat actually never touches Ollama — it runs on `claude-3-5-haiku-latest` through `chat-proxy`'s own `MODEL_MAP` (the `modelKey` sent is the literal string `'free-default'`, not the `model` variable). Every free-tier analytics event will report `llama3:8b` as the model used, when the real model is Claude Haiku. Product/usage analytics for your largest user segment will be wrong until this is fixed.

### 🟡 MEDIUM — Edge cases, error handling, UI/UX alignment, code stability

**1. Free tier's own voice error message tells users to upgrade to a tier where voice is equally broken.**
`src/main/voice.ts`: when a free-tier user hits the local-Whisper requirement, they're told *"Voice transcription requires a Pro subscription for cloud accuracy."* Per 🔴 URGENT #1, Pro also cannot use voice (same missing key/proxy). Once #1 is fixed this resolves itself, but until then this message actively steers users toward paying for something that won't fix their problem.

**2. Stale/incorrect comment on the usage-tracking migration.**
`supabase/migrations/001_create_usage_tracking.sql` says *"Free = 20/day on OUR API keys"* in a comment, but the two places that actually enforce the limit — `supabase/functions/chat-proxy/index.ts`'s `DAILY_LIMIT` map and `src/main/stripe/pricing.ts`'s `TIERS.free.dailyMessageLimit` — both agree on **5**/day. The enforcement is internally consistent (not a functional bug), but the stale comment will mislead the next person who edits this file.

**3. `mcp-config.json` / the `openui:mcp:connect` IPC handler has no matching UI and no input validation.**
`src/main/index.ts` registers `ipcMain.handle('openui:mcp:connect', ...)` which accepts an arbitrary `config: unknown` and passes it straight to `connectMcpServer()` in `src/main/mcp-client.ts` — for a `stdio` transport this ultimately spawns `config.command` with `config.args` as a local child process, with no allowlist. Searching the entire renderer for any reference to "mcp" turns up nothing — there is no Settings panel or UI of any kind that lets a real user reach this handler. As shipped, it's an unreachable capability; if a UI is added later, the config should be validated/allowlisted before it reaches `connectMcpServer`.

**4. Voice usage metering is a byte-size heuristic, not real duration.**
`src/main/voice.ts`: `const durationSeconds = Math.round(audioBuffer.byteLength / 12000)` estimates spoken minutes from compressed file size rather than decoding actual audio duration. Depending on codec/bitrate variance across browsers/OSes, this will over- or under-count usage against the monthly voice-minute cap that's supposed to gate the Pro/Enterprise tiers.

**5. Inline styling throughout the renderer bypasses the Tailwind setup that's already configured.**
`tailwind.config.js` and `postcss.config.js` are present and wired into `electron.vite.config.ts`, yet components like `SettingsModal.tsx` and the "Workflows" button in `App.tsx` use large inline `style={{...}}` objects instead. Not a functional bug, but it's two competing styling systems in one small renderer, and it will only get harder to keep consistent as more UI ships.

**6. `server/` duplicates tier/pricing logic that already lives correctly in the shipped app.**
`server/tiers.py` re-defines the same tier limits already defined in `src/main/stripe/pricing.ts` and enforced in `chat-proxy`. The numbers currently agree, but three independent sources of truth for the same business rule (soon to be a fourth if `api/` is ever finished) is a standing risk of drift. Worth collapsing to one source if `server/` survives the 🔴 URGENT #8 decision.

**7. No in-app expectation-setting for the free-tier `read_screen` fallback.**
`read_screen()` in `tools.ts` silently degrades to Tesseract OCR text-only output for free-tier users, versus the coordinate-aware vision description Pro/Enterprise tiers are meant to get. The tool result even tells the *model* to consider recommending an upgrade, but there's no proactive UI messaging before the user hits this, so the gap in capability reads as a bug rather than a tier boundary the first time someone hits it.

Update the description/metadata to match what's actually shipping.

**2. Three separate dependency graphs live in one repo.** Root `requirements-api.txt` (for `api/`), `server/requirements.txt` (for `server/`), and the Electron `package.json` describe three unrelated sets of dependencies for what is, functionally, one shipping product plus two disconnected experiments. Resolving 🔴 URGENT #8 and 🟠 HIGH #3 will naturally collapse this.

**3. `admin-dashboard.html` (26 KB) and `design.html` (74 KB) sit at the repo root outside any build system.** It's not clear from the surrounding code whether these are live internal tools, static design references, or abandoned prototypes. Worth a one-line note in the repo (or relocating into `/docs`) so it's clear whether they need to be maintained.

**4. `CHANGELOG.md` is 761 bytes for a project at version `7.0.4`.** Release-notes discipline hasn't kept pace with the version number, which will show up directly in the auto-updater's "what's new" experience once `UpdateBanner`/`UpdateReady` surface release notes to users.

**5. The free-tier local-Whisper fallback (`WHISPER_CPP_PATH`) is a developer-only escape hatch.** It requires a self-compiled `whisper.cpp` binary with no installer or setup flow — realistically never set on a real user's machine. Once 🔴 URGENT #1 is fixed with a proper voice-proxy, consider dropping this path entirely rather than maintaining dead configuration surface.

## Summary

The shipped Electron app's core loop — sign in, chat, let the model safely take actions on your machine, get billed correctly — is built by someone who clearly understands security and UX tradeoffs, and it would hold up well under real usage as-is. The rating is pulled down hard by three things stacked together: several named, paid-tier features are completely non-functional for every real user due to a missing proxy-layer pattern that chat itself proves the team knows how to build (🔴 #1–#5); a real broken-access-control bug sits on billing endpoints (🔴 #6); and the repository carries enough dead/duplicate/orphaned code (🔴 #7–#8, 🟠 #3–#5) that "what is actually running in production" is not obvious from reading the repo alone. None of the 🔴 items are architecturally hard to fix — #1 through #5 are all the same fix applied five times (add a server-side proxy or a personal-token field, matching the pattern `chat-proxy` already established) — but until they're fixed, this is not yet a product you can put in front of paying users without them immediately hitting broken, advertised functionality.
